## Overview

This repository contains the source for the **A8n documentation website** — built as a statically exportable Next.js 15 application powered by [Nextra 4](https://nextra.site). The site covers:

| Section | Description |
|---|---|
| **Getting Started** | Platform overview and first-deployment walkthrough |
| **Features** | Core features — CI/CD, auto-scaling, GitOps, monitoring, secrets, testing |
| **Authentication** | OAuth login, role-based access control, and account management |
| **Deployments** | Monoliths, microservices, databases, Helm + ArgoCD GitOps, rollbacks |
| **Monitoring** | Real-time metrics, log streaming, alerts, and health checks |
| **Security** | Vault secrets, RBAC, image scanning, network isolation, TLS |
| **CLI Reference** | Full command reference for the `a8s` CLI tool |
| **Storage & Billing** | Quota management, PVC billing, and usage monitoring |

The site uses [Pagefind](https://pagefind.app) for fully offline, zero-dependency full-text search built at deploy time.

## Getting Started

### Prerequisites

- **Node.js** 18 or later
- **npm** 9+ (or pnpm / yarn)

### Install & Run

```bash
# 1. Clone the repository
git clone https://github.com/Vuthy-Tourn/a8s-docs.git
cd a8s-documentation

# 2. Install dependencies
npm install

# 3. Start the development server
npm run dev
```

Open [http://localhost:3000](http://localhost:3000) in your browser. The site auto-redirects to `/docs`.

### Build for Production

```bash
# Build the Next.js app
npm run build

# (Optional) Run Pagefind to regenerate the search index
npx pagefind --site .next --output-path public/_pagefind

# Start the production server
npm start
```

> **Note:** The `public/_pagefind/` directory is generated at build time and should **not** be edited manually. It is committed to the repo so the search index is always available without a separate build step in CI.

## Content Authoring

All documentation lives in `src/content/` as `.mdx` files. You can use standard Markdown, HTML, and any component imported from `nextra/components` or your own `src/components/`.

### Adding a New Page

1. Create a new `.mdx` file in the appropriate `src/content/` subfolder:

```bash
# Example: add a new "Scaling" page under Features
touch src/content/features/scaling.mdx
```

2. Add frontmatter at the top of the file:

```mdx
---
title: Auto-Scaling
description: Configure HPA rules and observe scale events in real time.
---
```

3. Register the page in the section's `_meta.ts` file:

```ts
// src/content/features/_meta.ts
export default {
  'core-features':         { title: 'Core Features' },
  'scaling':               { title: 'Auto-Scaling' },   // ← add this
  'authentication':        { title: 'Authentication' },
  // ...
}
```

The new page will appear in the sidebar automatically.

### Available Nextra Components

These components can be imported directly in any `.mdx` file:

```mdx
import { Callout, Steps, Cards, Card, Tabs, Tab, FileTree } from 'nextra/components'
```

| Component | Use |
|---|---|
| `<Callout type="info|warning|error">` | Highlighted info boxes |
| `<Steps>` | Numbered step-by-step guides |
| `<Cards> / <Card>` | Navigation card grids |
| `<Tabs> / <Tab>` | Tabbed code or content blocks |
| `<FileTree>` | Repository tree diagrams |

## Configuration

### `next.config.ts`

```ts
import nextra from 'nextra'

const withNextra = nextra({
  contentDirBasePath: '/docs',
})

export default withNextra({ reactStrictMode: true })
```

### `tailwind.config.ts`

Tailwind 4 uses a CSS-first `@theme` block in `globals.css` — `tailwind.config.ts` is minimal and primarily used for the `content` paths:

```ts
export default {
  content: ['./src/**/*.{ts,tsx,mdx}'],
}
```

The full design token set (fonts, colors, spacing) is defined in `src/app/globals.css` via `@theme { … }`.

### `src/app/docs/[[...mdxPath]]/page.jsx`

This is Nextra 3's dynamic catch-all route. It resolves every `/docs/…` URL to its corresponding MDX file under `src/content/`. **Do not rename or move this file.**

## Contributing

Contributions — corrections, new pages, example improvements — are welcome.

1. **Fork** this repository and create a branch: `git checkout -b docs/your-topic`
2. Make changes to files under `src/content/`
3. Run `npm run dev` and verify your changes locally
4. Run `npm run type-check` and `npm run lint` — both must pass
5. Open a **Pull Request** with a clear description of what changed and why

### Writing Guidelines

- Use sentence case for headings (not Title Case every word)
- Keep paragraphs short — 2–4 sentences max
- Prefer bullet lists over walls of text for feature descriptions
- Always include a frontmatter `title` and `description` in every MDX file
- Use `<Callout type="warning">` for anything that could cause data loss or deployment failure
- Code examples should be complete, runnable, and include the language identifier on the fence
